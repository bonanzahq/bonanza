# Rails 8.1.2 + Ruby 4.0.1 Upgrade Session - 2026-02-14

## Goal

Upgrade Rails 8.0.4 -> 8.1.2 and Ruby 3.4.8 -> 4.0.1, as planned in the
previous session's next-steps.

## What we did

### Rails 8.0.4 -> 8.1.2

- `bundle update rails` resolved cleanly. No gem conflicts -- acts-as-taggable-on
  13.0.0 requires `activerecord >= 7.1` with no upper bound.
- Tests: 200 runs, 0 failures, 2 pre-existing ES errors.
- Committed separately from load_defaults bump.

### load_defaults 8.0 -> 8.1

Reviewed Rails 8.1 defaults:
- `yjit = !Rails.env.local?` -- YJIT in production only
- `escape_json_responses = false` -- no HTML entity escaping in JSON
- `action_on_path_relative_redirect = :raise` -- checked all controllers, all use named routes
- `raise_on_missing_required_finder_order_columns = true` -- stricter ordering
- `remove_hidden_field_autocomplete = true` -- cosmetic HTML change

Tests pass with no issues on 8.1 defaults.

### Ruby 3.4.8 -> 4.0.1

- `mise install ruby@4.0.1` succeeded
- Updated Gemfile, mise.toml, Dockerfile
- **minitest 5.25.4 blocked**: had `required_ruby_version < 4.0`. Bumped to 5.27.0
  which drops the upper bound.
- `bundle install` succeeded, all native gems compiled fine (bcrypt, pg, puma,
  nokogiri, oj, redcarpet)
- Tests: 200 runs, 0 failures, 2 pre-existing ES errors
- No frozen string literal issues -- Ruby 4.0 didn't make them default after all
- Docker build succeeded, app boots and serves /login (200 OK)
- Bundler 2.6.9 emits platform constant redefinition warnings against Ruby 4.0.0
  stdlib -- cosmetic, no functional impact

## Technical insights

- minitest 5.26.1+ is the first version to remove the `< 4.0` Ruby constraint.
  5.27.0 is the latest 5.x line.
- Ruby 4.0.1 Docker image exists on Docker Hub (released 2026-01-13)
- acts-as-taggable-on 13.0.0 has no upper bound on activerecord, making Rails
  minor upgrades painless
- The app's Devise routes use `/login` not `/users/sign_in` -- custom path mapping
