### 16. CI/CD Deployment Planning
- Created comprehensive plan at `docs/plans/ci-cd-deployment.md`
- **Deployment architecture**: Docker Compose + SSH (KISS principle)
  - No orchestration overhead (no Swarm, Kubernetes, Portainer)
  - Single app per server, no need for complex tooling
- **CI/CD workflow**:
  - PR → Run tests + test Docker build
  - Push to main → Build :beta tag → Deploy to beta server
  - Push git tag (v1.2.3) → Build :latest + :v1.2.3 tags → Deploy to production
- **Registry**: GitHub Container Registry (GHCR)
  - Free private repos, seamless GitHub Actions integration
- **Versioning**: Manual git tags (semantic versioning)
  - Can add semantic-release automation later if needed
- **Deployment script**: Fault-tolerant bash script
  - Uses existing image if registry pull fails
  - Runs migrations automatically
  - Reindexes Elasticsearch
  - Waits for health checks
  - Rolls back on failure
  - Cleans up old images
- Three GitHub Actions workflows planned:
  1. `test.yml` - Test on PRs (Minitest + Docker build)
  2. `deploy-beta.yml` - Auto-deploy beta on main branch
  3. `release.yml` - Release production on git tags
- Server setup: Ubuntu + Docker + docker-compose
  - Simple directory structure: `/opt/bonanza`
  - Environment secrets in `.env` file
  - SSH key-based deployment
- Monitoring: Docker logs + health checks
- Backup strategy: Automated database backups, volume backups
- Rollback procedures: Documented and tested
- Alternative considered: Kamal (37signals' Rails deployment tool)
  - Worth evaluating for maintained tooling vs custom scripts
